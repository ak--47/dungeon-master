## Wave 1 Learnings (from sass.js — pre-existing)

Documented in FINALIZE-HOOKS.md as L1-L7. Key patterns:
- L1: Temporal hooks belong in `everything`, not `event`
- L2: Event cloning requires `everything` hook (push, not return)
- L3: Property baseline must contrast with hook target
- L9: DuckDB anchor = `TIMESTAMP '2026-01-01'`

## Wave 2 Learnings

1. **Agentic cohort sizing** (ai-platform): When detection condition catches >90% of users (1+ tool_use is nearly universal), the cohort has no meaningful control group. Fix: require 3+ events for cohort membership.

2. **Clone dilution of temporal effects** (crypto, real-estate): Hooks that clone events with time offsets can place clones inside temporal windows (gas spike, spring season). The temporal mutation must run AFTER all cloning to catch these events. Pattern: move temporal value mutations to the end of the everything hook.

3. **Cross-hook event volume confounds** (crypto rug-pull): When a behavioral cohort (SCAM holders) correlates with activity level, raw per-user metrics are misleading. High-activity users have more events even after 80% removal. Fix: use normalized metrics (events per activity unit) or reduce cohort membership rate.

4. **Unseeded chance instances** (ecommerce): Some 1.3.1-era dungeons used `new Chance()` without the seed, producing non-deterministic output. Fix: replace with `initChance(SEED)`.

5. **Event hook `meta.datasetEnd` unreliable** (ecommerce): Same as L1 but for datasetEnd — event hook timestamps are pre-shift while meta timestamps may be post-shift. All temporal checks must be in `everything`.

## Wave 3 Learnings

6. **Deprecated feature replacement** (fitness, devtools, healthcare): Dungeons using deprecated `subscription`, `attribution`, `features` config blocks had hooks that depended on properties generated by those features. The engine silently strips the config, so the properties never appear. Fix: add equivalent property assignments in the hook (user or everything) or add the property to superProps/userProps.

7. **Magic number bucket sizing** (dating, food-delivery): With low event rates (avg <1 event/day), natural per-user event counts are low. Sweet spot ranges like "5-8" may capture <50 users. Fix: lower thresholds to match actual distribution or increase population.

8. **Event hook for non-temporal mutations is fine** (all): Closure-based Maps (cost overrun → scale down), event replacement (alert → incident), and simple property mutations work correctly in the event hook. Only temporal checks and event cloning need `everything`.

9. **Funnel-post TTC not SQL-verifiable by design**: All 20 dungeons have a funnel-post TTC hook. These are rated STRONG by mechanism inspection — the code path is verified correct but cross-event SQL can't reproduce Mixpanel's within-funnel TTC calculation.

10. **Batch mode glob pattern**: Dungeons with >2M events auto-batch. DuckDB queries must use `read_json_auto('./data/verify-X-EVENTS-part-*.json', sample_size=-1, union_by_name=true)`.
